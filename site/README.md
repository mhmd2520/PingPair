# PingPair website (`pingpair.app`)

The marketing site served at **https://www.pingpair.app** — a plain static
site (hand-written HTML/CSS, no build step). This is **Phase W** of the
roadmap; see `plan.md` → "Phase W — Marketing website" for the full design and
sub-phase plan.

## What's here

```
site/
├── index.html        # landing page (hero, stats, features, how-it-works, gallery, download, trust, footer)
├── docs.html         # W4 — install / setup / run / read-the-report / troubleshooting / FAQ
├── changelog.html    # W5 — release notes (kept in sync with /CHANGELOG.md)
├── 404.html          # W5 — not-found page (GitHub Pages serves it automatically)
├── styles.css        # "Signal Instrument" design system — mirrors the app's Dark theme + logo palette
├── app.js            # progressive enhancement only: scroll-reveal + sticky-nav + gallery Light/Dark toggle + lightbox
├── CNAME             # custom domain for GitHub Pages: www.pingpair.app
├── robots.txt        # allow-all + sitemap pointer
├── sitemap.xml       # W5 — home / docs / changelog
└── assets/
    ├── logo.svg      # the app mark (copied from Software/src/pingpair/resources/logo.svg)
    ├── fonts/        # bundled Inter (same fonts the app ships — no external CDN)
    ├── img/
    │   ├── hero-run.png         # hero screenshot (Run tab, Dark)
    │   ├── tour/{dark,light}/   # W3 gallery — 6 tab renders per theme
    │   └── diagrams/            # W4 — topology / loopback / report-artifacts (from build_help_diagrams)
    └── sample/                  # W3 — downloadable 20-case sample report (.docx + .pdf)
```

The brand palette, fonts and logo are copied from the app so the site and the
app read as one product. The screenshots under `assets/img/tour/` and the
diagrams under `assets/img/diagrams/` are copied from
`Software/src/pingpair/resources/help/` (`_shots/<theme>/client/<tab>/` and
`_assets/<theme>/`); **re-copy them after any Help/welcome screenshot re-shoot**
so the site shows the current UI.

The **sample report** under `assets/sample/` (`PingPair-sample-report.{docx,pdf}`)
was generated by the real reporting pipeline (so it carries the report-header
logo + analysis appendix) from synthetic-but-plausible 20-case data — **not** a
committed dev run (all of those are 1-case). Regenerate it after any
report-format change by feeding a `RunReport` of 20 `CaseMetrics` to
`reporting.docx_writer.write_docx` / `pdf_writer.write_pdf` with
`include_appendix=True` (run under the venv with `QT_QPA_PLATFORM=offscreen` so
the logo rasterizes headless).

> **The page is usable with JavaScript off.** `app.js` only adds enhancements —
> scroll-reveal (gated behind a `js` class on `<html>`, so content is fully
> visible without JS), the sticky-header scrolled state, the gallery Light/Dark
> toggle, and the click-to-zoom lightbox. With JS disabled the gallery shows the
> Dark screenshots and every link/download still works. All motion is disabled
> under `prefers-reduced-motion`.

## Keeping the download page current (W6)

The download page carries machine-updatable sentinel markers
(`<!--@ver-->…<!--/@ver-->`, `<!--@size-->`, `<!--@sha256-->`) plus two anchors
(`id="dl-asset"`, `id="dl-checksum-file"`) whose hrefs are rebuilt from the
version. **`publish/sync-site-release.ps1`** rewrites all of them:

```powershell
pwsh -File publish\sync-site-release.ps1 -Version 0.1.0 `
     -Sha256File Software\dist\PingPair-0.1.0-win64.zip.sha256 -SizeBytes 104809496
```

This runs **PrivDev-side, inside `/publish`**, right after the bundle's SHA-256
is known and **before** `export-to-public.ps1` copies `site/` to the public
repo. A GitHub Action committing these values into the public repo would be the
"obvious" W6 design, but the allowlist export `/MIR`s `site/` over the public
checkout — so any such commit would be clobbered on the next publish. The
source of truth is `site/` here in PrivDev; updating it before the export is the
only design that stays consistent.

## Preview locally

From this `site/` folder:

```powershell
py -m http.server 8000
```

then open <http://localhost:8000>. (A plain `file://` open mostly works too,
but a local server avoids font/asset path quirks.)

## Go live — deploy checklist

Hosting = **GitHub Pages** on the public repo `mhmd2520/PingPair`; the domain
is registered at **Hostinger (DNS only)**. The `.exe` stays on GitHub Releases —
the site never serves it.

### 1. Get `/site` into the public repo — DONE (in PrivDev)
`publish/export-to-public.ps1` already lists both `site` and `.github` on the
allowlist (`$IncludeDirs = @('Software', 'site', '.github')`), and the Pages
**deploy workflow** lives at `.github/workflows/deploy-site.yml` here in the dev
tree. The next allowlist export (standalone, or via `/publish`) carries both
into the public repo. The workflow's deploy job is gated on
`github.repository == 'mhmd2520/PingPair'`, so it's inert on the PrivDev mirror
that also carries the file (PrivDev has no Pages site).

(GitHub Pages' built-in "deploy from branch" only serves repo-root or `/docs`,
so the site — which lives in `site/` — is deployed via Actions instead.)

To push the site into the public repo without cutting a full release, run the
export standalone and commit the public checkout:

```powershell
pwsh -File publish\export-to-public.ps1            # mirrors Software + site + .github -> D:\PingPair
git -C D:\PingPair add -A
git -C D:\PingPair commit -m "site: deploy pingpair.app (Phase W)"
git -C D:\PingPair push
```

### 2. Enable Pages
Public repo → **Settings → Pages** → Source = **GitHub Actions**. After the
first deploy, set **Custom domain = `www.pingpair.app`** and tick **Enforce
HTTPS** (the `.app` TLD is HSTS-preloaded — HTTPS is mandatory; the cert is
issued automatically and can take a few hours).

### 3. DNS at Hostinger (zone editor for `pingpair.app`)
Delete any default parking `A`/`CNAME` record on `@` first, then add:

| Type  | Name  | Value                  |
|-------|-------|------------------------|
| CNAME | www   | `mhmd2520.github.io`   |
| A     | @     | `185.199.108.153`      |
| A     | @     | `185.199.109.153`      |
| A     | @     | `185.199.110.153`      |
| A     | @     | `185.199.111.153`      |
| AAAA  | @     | `2606:50c0:8000::153`  |
| AAAA  | @     | `2606:50c0:8001::153`  |
| AAAA  | @     | `2606:50c0:8002::153`  |
| AAAA  | @     | `2606:50c0:8003::153`  |

`www` is the primary (matches `CNAME`); the apex `A`/`AAAA` records let
`pingpair.app` redirect to `www.pingpair.app`. (GitHub's Pages IPs above are
current as of 2026-06; confirm against GitHub's "Managing a custom domain"
docs if they ever change.)

### 4. Optional — email
Set up free **email forwarding** at Hostinger (`hello@pingpair.app` → your
Gmail) for the footer contact link.
