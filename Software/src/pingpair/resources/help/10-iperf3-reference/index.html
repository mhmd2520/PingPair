iperf3 3.21 — flag reference

PingPair ships iperf3 3.21 (Cygwin build) under bin\iperf3\. The flags below are the ones PingPair sets on either the server or the client side per case.

Server invocation (auto-spawned per case)

iperf3 -s -i 1 -B 192.168.1.1 -1 --json

Client invocation (per case)

iperf3 -c 192.168.1.1 -B 192.168.1.2 -u -i 1 -t 30 -l <payload> -b <bw>M

Flags used

FlagMeaningWhy PingPair uses it
-s Run as server (listen for an incoming client). One side per case must listen.
-c <host> Run as client and connect to host. Drives the actual data transfer for the case.
-B <ip> Bind to a specific local IP for sending/receiving. Pins the Server to 192.168.1.1 and the Client source to 192.168.1.2. Dropped in loopback mode where both endpoints are 127.0.0.1.
-u Use UDP (not TCP). The procedure mandates UDP — gives measurable jitter and packet-loss numbers, which TCP hides via retransmission.
-i <sec> Print interval reports every sec seconds. 1-second intervals — used by the live throughput chart.
-t <sec> Time the test for sec seconds. 30 s per case (the procedure value).
-l <bytes> Set the read/write buffer size = the UDP datagram payload. Per-case payload from the 4-value sweep: 200, 600, 1000, 1300 bytes.
-b <rate>M Cap the sender's bitrate at rate Mbits/s. Per-case bandwidth from the 5-value sweep: 10, 30, 50, 70, 90 Mbps.
--json Print all output as one JSON blob at end of run instead of text. Used on the Server side so the JSON can be archived in the report sidecar. The Client side uses text mode so live per-second output appears on the chart.
-1 Server: handle one client connection then exit. Avoids stale listening sockets between cases — without this, a force-terminated server can leave port 5201 half-closed and the next case fails with "unable to receive control message".
