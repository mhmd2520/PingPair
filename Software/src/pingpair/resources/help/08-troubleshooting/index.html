Troubleshooting

Each entry below is symptom → root cause → fix. Most fixes are a single button on the Setup tab.

The Client can't reach the Server / "control channel timed out"

Root cause & fix — work through these in order:

Likely causeFix
The control-channel firewall rule is missing on the Server. On the Server, Setup tab → Open 5202 (TCP).
The Server's IP (192.168.1.1) isn't bound yet, so its listener is still waiting. On the Server, Setup tab → Set the correct IP. The listener picks it up automatically — no restart.
The Server side isn't running PingPair, or isn't in Server role. Launch PingPair on the Server PC and confirm the top banner says Server.

A plain ping can succeed (ICMP) while the control channel still fails — they use different ports. Confirm port 5202 is open, not just that ping works.

Both PCs show a 169.254.x.x address

Root cause: the isolated Ethernet link has no DHCP server, so each laptop fell back to an APIPA self-assigned address (169.254.x.x) and never got 192.168.1.x.
Fix: on each PC, Setup tab → Set the correct IP. That assigns the role's static IP via netsh. The Local NIC IP check turns green and the Server listener comes up on its own.

Orange banner: "this PC's IP doesn't match the saved role"

Root cause: the role saved for this PC expects an IP that isn't currently bound (NIC reset, moved LAN, DHCP reclaim).
Fix: either click Set the correct IP to bind the expected address, or change the role with the Setup-tab Role selector to match what this PC actually is. PingPair never flips the role for you.

The orange banner sits across the top of every tab. It spells out the expected vs. bound IP, and its "How to fix" button brings you straight here. Click the image to enlarge.

Reports show packet loss that shouldn't be there

Root cause: a Wi-Fi adapter is online and carrying an IPv4 address, so Windows sometimes routes 192.168.1.x test traffic over Wi-Fi instead of the dedicated Ethernet link — producing phantom loss.
Fix: Setup tab → Disconnect Wi-Fi. It's re-enabled automatically when you close the app or click Reset all settings.

The fix buttons are greyed out

Root cause: the firewall and IP fixes use netsh, which needs Administrator. The app isn't elevated.
Fix: Restart as Administrator on the Admin row. On Windows PingPair normally auto-elevates at launch via a UAC prompt; if you declined it, this button re-triggers elevation.

"fping binary not found" / "iperf3 binary not found"

Root cause: the bundled executable is missing from bin\fping\ or bin\iperf3\.
Fix: restore the file from the project's Network-Tools\ folder. The two tools ship with different runtime DLLs and must stay in their own subfolders.

"Gateway reachable" shows a warning

Root cause: a gateway is configured but didn't answer a ping — usually a typo, or a gateway set on a point-to-point LAN that has none.
Fix: on the Config tab, correct the gateway IP, or clear the Gateway field if this PC is on a direct point-to-point link. This is a warning, not a blocker.

A sweep "hangs" at ~50 seconds per case

Not a bug. Windows' ~15.6 ms timer granularity stretches fping's 10 ms ping interval, so each case takes about 50 s rather than the nominal 30 s, and a full 20-case sweep runs ~16 minutes. Let it finish.
